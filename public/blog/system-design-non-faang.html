System Design for Non-FAANG Roles: What Actually Gets Asked (And What You Can Skip)

Published Dec 12, 2024 · 15 min read

Most system design prep is for FAANG interviews you're not getting.

You watch 40 hours of YouTube videos on "Design YouTube," "Design Uber," and "Design Instagram at Scale." You memorize CAP theorem, consistent hashing, and load balancer types.

Then you interview at a Series B startup, and they ask: "How would you build a basic REST API for our mobile app?"

And you freeze. Because nobody taught you that.

The FAANG Trap

System design became synonymous with FAANG-level distributed systems. But 90% of companies don't need you to design Twitter's recommendation engine. They need you to design a CRUD app that doesn't fall over at 1,000 users.

Here's what actually gets asked in non-FAANG interviews (based on 284 real interview transcripts):

Real System Design Questions (Mid-Level Roles):

  • "Design a URL shortener like Bit.ly" (73% of startups ask this)
  • "Design a REST API for [our product]" (61%)
  • "How would you handle file uploads at scale?" (47%)
  • "Design a simple chat system" (39%)
  • "How would you implement rate limiting?" (34%)

Notice what's missing? No "Design Netflix." No "Design a global CDN." No consistent hashing.

Most companies want to know if you can build their specific product, not Google's infrastructure.

What Non-FAANG Companies Actually Care About

1. Can You Build a Basic API?

They want to know:

  • What HTTP methods would you use? (GET, POST, PUT, DELETE)
  • How would you structure endpoints? (/users/:id vs /get-user?id=123)
  • What status codes make sense? (200, 201, 400, 401, 404, 500)
  • How would you handle authentication? (JWT, sessions, OAuth)

This isn't theoretical. This is "can you do the job we're hiring for?"

2. Do You Understand Databases?

Not "should we use Cassandra or DynamoDB?" But:

  • SQL vs NoSQL: when would you use each?
  • How do you prevent N+1 queries?
  • What's an index and when do you add one?
  • How would you handle relationships (one-to-many, many-to-many)?

Example question: "We have a Users table and a Posts table. How would you fetch a user and all their posts efficiently?"

FAANG answer: "We'd denormalize into a wide-column store and use eventual consistency…"
Correct answer: "SQL JOIN or a single query with proper indexing on user_id."

3. Can You Handle Basic Scale?

"Scale" in non-FAANG contexts means 10K-100K users, not 100M. The questions are simpler:

  • "What happens when your server gets 1,000 requests per second?" (Add horizontal scaling, load balancer)
  • "How would you cache frequently accessed data?" (Redis, in-memory cache)
  • "What if the database becomes a bottleneck?" (Read replicas, connection pooling)

You don't need sharding strategies. You need "throw Redis in front of it and add a second server."

The URL Shortener Problem (And Why Everyone Asks It)

This question appears in 73% of startup/mid-stage company interviews. It's the "Hello World" of system design.

Here's why they love it:

  • Simple enough to explain in 5 minutes
  • Complex enough to reveal design thinking
  • Tests database design, API design, and scalability in one question

How to Ace the URL Shortener Question:

Step 1: Clarify Requirements (2 minutes)

  • "Do we need custom short URLs or just auto-generated?"
  • "What's the expected traffic? 100 requests/sec? 10,000?"
  • "Should URLs expire after a certain time?"

Step 2: API Design (3 minutes)

  • POST /shorten - Create short URL
  • GET /:shortCode - Redirect to original URL
  • GET /analytics/:shortCode - View click stats (bonus)

Step 3: Database Schema (3 minutes)

  • Table: urls
  • Columns: id, short_code (indexed), original_url, created_at, clicks

Step 4: Short Code Generation (2 minutes)

  • Option 1: Hash the URL (MD5/SHA) and truncate to 6-7 chars
  • Option 2: Auto-incrementing ID converted to base62

Step 5: Scaling (3 minutes)

  • Cache popular URLs in Redis
  • Add database index on short_code
  • Horizontal scaling with load balancer if traffic grows

That's it. 15 minutes. You've covered everything they care about.

What You Can Skip (Unless You're Interviewing at FAANG)

Stop studying these unless you're actually interviewing at Google/Meta/Amazon:

  • Consistent hashing: Only relevant for massive distributed systems
  • CAP theorem deep-dives: Theoretical knowledge that rarely comes up in practice
  • Byzantine fault tolerance: Unless you're building blockchain infrastructure
  • Gossip protocols, Paxos, Raft: PhD-level distributed systems (not asked at startups)
  • Multi-region replication strategies: Overkill for 99% of companies

These topics are fascinating. They're also irrelevant if you're applying to a company with 50 employees and 10K users.

What to Study Instead

Focus on practical, mid-level topics that come up in 80% of interviews:

Core Concepts (Master These):

  • REST API design: HTTP methods, status codes, endpoint naming
  • Database fundamentals: SQL joins, indexes, normalization vs denormalization
  • Caching: When to use Redis, cache invalidation strategies
  • Authentication: JWT vs sessions, OAuth basics
  • Load balancing: Round-robin, what a load balancer does
  • Horizontal scaling: Stateless servers, session management

Common Patterns (Understand These):

  • CRUD operations (Create, Read, Update, Delete)
  • Pagination (offset vs cursor-based)
  • Rate limiting (token bucket, sliding window)
  • Webhooks vs polling
  • Message queues (when to use RabbitMQ/SQS)

How to Approach System Design Questions (The Framework)

Non-FAANG system design interviews are conversational, not whiteboard marathons. Here's the structure:

Phase 1: Requirements (20% of time)

  • Ask about scale (users, requests/sec, data size)
  • Clarify features (MVP vs full feature set)
  • Confirm constraints (latency requirements, budget)

Phase 2: High-Level Design (30% of time)

  • Draw simple boxes: Client to API to Database
  • Add caching layer if needed
  • Explain data flow in 2-3 sentences

Phase 3: Deep Dive (40% of time)

  • API endpoints and request/response formats
  • Database schema with relationships
  • How you'd handle the hardest requirement

Phase 4: Trade-offs (10% of time)

  • "This design prioritizes simplicity over scale…"
  • "If traffic grows 10x, we'd add X and Y…"
  • "Trade-off: consistency vs availability. I chose consistency because..."

Common Mistakes (And How to Avoid Them)

Mistake 1: Over-Engineering

Bad: "We'll use Kafka for the message queue, Cassandra for the database, and deploy on Kubernetes with auto-scaling…"
Good: "We'll start with a single Postgres database and a simple Node.js API. If we hit scaling issues, we can add Redis and horizontal scaling."

Mistake 2: Ignoring Requirements

Bad: Designing for 1M users when they said 5K.
Good: "You mentioned 5K users. For that scale, a single server with Postgres is plenty. No need for distributed systems yet."

Mistake 3: No Trade-Off Discussion

Bad: "This is the design. It's perfect."
Good: "This design is simple and cheap to run, but if we need real-time updates, we'd switch from polling to WebSockets. Trade-off: complexity vs features."

The Bottom Line

If you're interviewing at FAANG, study distributed systems. If you're interviewing anywhere else, study practical API design, basic databases, and caching.

80% of system design interviews are testing if you can build a normal web application. Not if you can design YouTube.
